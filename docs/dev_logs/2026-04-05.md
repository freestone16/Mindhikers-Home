# 2026-04-05

## 会话概览

- 当前目标：评估并安装 `gstack` 与 `CE` 两套全局 workflow / skill 体系
- 决策重点：
  - 是否适合全局安装
  - 安装后是否会造成 skill 过载
  - 如何收敛成一套可长期使用的组合

## 本轮结论

1. 两套方案都适合做“全局安装”，但不适合都作为默认主流程同时全开
2. 最终采用组合策略：
   - `CE` 作为主工作流
   - `gstack` 作为补强层，只保留最有价值的 review / QA / challenge 模块
3. 未直接删除多余 `gstack` skill，而是移动到全局 disabled 目录，便于后续恢复

## 为什么这样装

### 1. `CE` 的角色定位

1. `CE` 更适合作为主线流程，覆盖：
   - `ce-brainstorm`
   - `ce-ideate`
   - `ce-plan`
   - `ce-work`
   - `ce-review`
   - `ce-compound`
2. 它的核心价值不是单个命令，而是：
   - 规划更深
   - 评审更系统
   - 能把经验沉淀到项目内文档，而不是只留在 session 里
3. 对当前仓库来说，更适合作为默认工作方式

### 2. `gstack` 的角色定位

1. `gstack` 原始设计就是“整套虚拟团队”，不是一个单点 skill
2. 它适合全局安装，但不建议在当前阶段把整套都作为常驻默认能力
3. 在当前组合里，`gstack` 的最高价值主要来自：
   - `gstack-plan-ceo-review`
   - `gstack-plan-eng-review`
   - `gstack-qa`
   - `gstack-qa-only`
   - `gstack-review`
4. 这几个模块刚好补齐：
   - 高层 scope / ambition challenge
   - 工程方案二次审查
   - 浏览器真实 QA
   - 额外代码 reviewer

## 当前推荐使用约定

### 1. 默认主流程

1. 需求探索：
   - `ce-brainstorm`
   - `ce-ideate`
2. 技术方案：
   - `ce-plan`
3. 实际执行：
   - `ce-work`
4. 审查：
   - `ce-review`
5. 经验沉淀：
   - `ce-compound`

### 2. `gstack` 只在这些场景补位

1. 需求是否值得扩大或重构方向：
   - `gstack-plan-ceo-review`
2. 技术方案需要额外工程审查：
   - `gstack-plan-eng-review`
3. 页面 / staging / 线上交互核验：
   - `gstack-qa`
   - `gstack-qa-only`
4. 代码修改后想再加一轮 reviewer：
   - `gstack-review`

## 初始化与后续动作建议

1. 重启 Codex 一次，让最新 active / disabled skill 状态稳定生效
2. 首次使用 `gstack` 时，如出现初始化问题，建议倾向：
   - telemetry：按个人偏好选择
   - proactive：建议先关闭，减少主动打断
   - routing rules：当前阶段先不要写入项目说明，避免和现有仓库规则冲突
3. 当前仓库已有明确规则：
   - 页面查看、UI 验证、截图、交互检查、线上页面核验优先使用 `agent browser`
4. 因此后续如 `gstack` 试图接管默认浏览器路由，需要谨慎，不要直接覆盖仓库既有约束

## 恢复策略

1. 本次不是删除，而是移动
2. 如需恢复任意 `gstack` 模块，只需从：
   - `~/.codex/skills-disabled/`
   - 移回 `~/.codex/skills/`
3. 恢复后重启 Codex 即可生效

## 本轮最终判断

1. 全局安装本身是合理的
2. 直接全量常驻使用偏重
3. 当前“`CE` 主线 + 精简 `gstack` 补强”的状态，是更适合本仓和当前阶段的平衡方案
