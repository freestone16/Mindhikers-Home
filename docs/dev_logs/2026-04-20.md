## 破局关键：m1-rest 插件部署

### 背景回顾

04-19 状态：仓库里有 `wordpress/mu-plugins/m1-rest/`（5 个 PHP 文件），但 staging WP 容器里没有，导致 `/wp-json/mindhikers/v1/blog` 返回 404。04-19 的 HANDOFF 文档列了 7 种失败尝试（Railway CLI ssh、GitHub raw、git clone、Web Shell 等），全部被容器封闭性拦住。

### 本轮解法

放弃"把文件塞进运行中容器"思路，改走 **WordPress 官方插件安装通道**：
1. 把 `m1-rest/` 目录里 5 个文件 + 新造的 `m1-rest.php` 主入口（带 Plugin Header + 路由注册逻辑）打包成标准插件 ZIP
2. 从 WP Admin 「插件 → 上传插件」安装并启用

### 中途踩的坑：函数名冲突

**v1.0.0 上传启用后路由仍 404。**

诊断方法：访问 `https://wordpress-l1ta-staging.up.railway.app/wp-json/`，看 `"namespaces"` 字段。结果找不到 `mindhikers/v1`，但找得到 `carbon-fields/v1` —— 说明旧版 `mindhikers-m1-core.php` 在跑，Carbon Fields 加载正常。

根因：v1.0.0 写了一段"防冲突闸门"
```php
if (function_exists('m1_register_rest_routes')) return;
```
这个 guard 被触发 —— 说明远端旧版 `mindhikers-m1-core.php` **确实定义了同名函数 `m1_register_rest_routes`**（哪怕是空壳或旧实现）。我预设"旧版不含 REST 路由"=> "不含同名函数"，这个跳跃是判断失误。

**v1.0.1 修复**：
- 移除 guard
- 函数名重命名为独一无二的 `mh_m1rest_standalone_register_routes`
→ 启用后路由正常注册，`/wp-json/mindhikers/v1/blog` 返回 200

## 本轮教训（待汇入 rules.md）

### Lesson 1：WP 插件静默失效的排查顺序

遇到"插件已启用但功能没生效"时，按以下顺序诊断，别一上来怀疑 fatal error：

1. **先查命名空间**：访问 `/wp-json/` 搜目标命名空间（如 `mindhikers/v1`）
   - 有 → 路由注册成功，往下查路径拼写
   - 无 → `add_action` 没挂上或 callback 被跳过
2. **再查函数名冲突**：WP 一个进程里函数名是全局单例，旧插件先声明了同名函数 → 新插件的 `add_action` 会调用到旧定义
3. **再查 PHP 版本**：PHP 8.0 语法（union 返回类型 `A|B`、`readonly`、`never` 等）在 7.4 下会 fatal；但 fatal 会触发 WP 自动停用插件，状态栏会标红
4. **最后查硬冲突**：`function_exists` guard、同命名空间 `register_rest_route` 后注册的会覆盖先注册的

**教训**：写"防冲突闸门"时，前提是你**清楚预期冲突的函数**是什么。预设"旧版不含 X" → "旧版没有 X 的符号"这个跳跃不成立。更稳妥的防冲突策略是**给自己的符号加独特前缀**（`mh_m1rest_*`），而不是靠检查对方。

### Lesson 2：敏感凭证轮换纪律

会话中出现过的 secret、token、API key，即使只出现一次也视为已泄漏。操作流程：
1. 验证结束后**立即在 Railway / 密钥库轮换**
2. 轮换后把新值同步到所有消费方（WP Carbon Fields、CI、本地 .env）
3. 不在 commit、PR、聊天记录里留原值
