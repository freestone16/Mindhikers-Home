### 2. “乱码”问题的根因与正式修复

之前看到的现象：

1. 浏览器直接打开 `wp-json/mindhikers/v1/homepage/zh`
2. 中文字段呈现为 `\\uXXXX`
3. 这会被误判成 CMS 内容或编码链路没有跑通

根因判断：

1. 根因不在 Next.js，也不在浏览器编码
2. 根因在 WordPress 自定义 REST 响应仍沿用默认 JSON 转义输出
3. 对中文内容来说，这会让自定义接口表现成“可机器读取，但不符合人工检查预期”

本次正式修复：

1. 在 `mindhikers-cms-core` 中加入 `rest_pre_serve_request` 过滤
2. 仅对 `mindhikers/v1/*` 路由改为：
   - `JSON_UNESCAPED_UNICODE`
   - `JSON_UNESCAPED_SLASHES`
3. 明确输出 `application/json; charset=UTF-8`

结果：

1. 线上 `zh` / `en` 接口现在已直接返回正常中文
2. 该修复发生在 CMS 输出层，不是前台补丁

## 2026-04-02 凌晨继续：根路径跳转问题的根因诊断

本轮新增确认：

1. 当前剩下的唯一问题已经不是 Access，也不是 DNS
2. 真正剩下的是：
   - `https://homepage-manage.mindhikers.com/`
   - 根路径 `/`
   - 仍返回 `200`
   - 仍显示默认 WordPress 前台页

本轮先做了“是否存在更底层根因”的核对，结论如下：

1. Railway 服务 `WordPress-L1ta` 的 `WORDPRESS_CONFIG_EXTRA` 之前仍写着 Railway 临时域名：
   - `DOMAIN_CURRENT_SITE='wordpress-l1ta-production.up.railway.app'`
   - `WP_HOME='https://wordpress-l1ta-production.up.railway.app'`
   - `WP_SITEURL='https://wordpress-l1ta-production.up.railway.app'`
2. 这会导致：
   - WordPress 生成的后台跳转残留临时域名
   - REST `_links` 残留临时域名
   - 站点内部“真身 URL”与正式域名不一致

本轮已执行修复：

1. 已将 `WORDPRESS_CONFIG_EXTRA` 改为正式域名版本：
   - `DOMAIN_CURRENT_SITE='homepage-manage.mindhikers.com'`
   - `WP_HOME='https://homepage-manage.mindhikers.com'`
   - `WP_SITEURL='https://homepage-manage.mindhikers.com'`
2. Railway 已完成一次成功重部署

修复后的验证结果：

1. `https://homepage-manage.mindhikers.com/wp-json/wp/v2/types`
   返回的 `_links`
2. 已从 Railway 临时域名切为：
   - `https://homepage-manage.mindhikers.com/...`
3. `https://homepage-manage.mindhikers.com/wp-admin/`
   仍正确进入 Cloudflare Access
4. 但 `https://homepage-manage.mindhikers.com/`
   仍返回 `200`

最新结论：

1. “站点正式 URL 真身未收口”这个根因已经解决
2. 但“根路径 `/` 是否应跳 `/wp-admin/`”是一个独立的入口策略问题
3. 它不会因为修正 `WP_HOME / WP_SITEURL` 自动消失

为什么在这里停手：

1. 下一步已经不再是排障，而是要在不同控制面之间做实现选择：
   - Cloudflare Redirect
   - MU Plugin redirect
   - 或其他入口治理方案
2. 当前 Cloudflare MCP 账号没有 Page Rules / Rulesets 写权限
3. 用户要求：如果这次还不能直接完成，就先停、完整落盘、交给外部专家诊断
