# MindHikers Lessons

更新时间：2026-03-30

## 2026-03-30：共享子域名与 Railway 模板初始化失配

### 现象

1. 在 `Mindhikers-Homepage` 项目为 Homepage CMS 规划管理域名后，隔壁产品出现了“访问自身域名却落到 Mindhikers CMS 管理域名”的异常线索。
2. 同时，`Primary` 这台 WordPress 服务虽然部署成功，但后台不可用：
   - `/` 返回 `403`
   - `/wp-login.php` 返回 `404`
   - `/wp-admin` 返回 `404`

### 从本项目侧确认到的事实

1. Homepage CMS 的最终管理域名已经收口为 `homepage-manage.mindhikers.com`。
2. 本仓库代码中没有发现把外部请求主动重定向到 Homepage CMS 管理域名的应用逻辑。
3. 旧现场里存在过 `cms.mindhikers.com` / 泛 `manage.mindhikers.com` 的过渡方案，但都不应再作为最终口径。
4. 当时的 CMS 服务返回的是 `403/404`，不是一个带 `Location` 的业务跳转页。

### 结论

1. 从 `Mindhikers` 这一侧看，没有证据表明这里主动把隔壁项目的流量重定向到了 Homepage CMS 管理域名。
2. 更可能的情况是：隔壁项目自身已经存在某种指向 Mindhikers CMS 管理域名的历史引用、反向代理、环境变量或业务跳转逻辑，而这次域名规划使该问题显性化。
3. 这次 `Mindhikers` 侧的域名变更，可能是“暴露问题的触发器”，但不像是“生成跳转逻辑的来源”。

### 同轮暴露出的第二个问题

1. 官方 `wordpress` 镜像依赖其首启时对 `/var/www/html` 的初始化逻辑。
2. 在未完全确认镜像行为前调整 volume 挂载路径，会把服务推入“不完整但非空”的目录状态。
3. 一旦进入这种状态，Apache 仍可启动，但 WordPress 核心文件可能缺失，导致后台不可用。

### 后续规则化动作

1. 共享根域下的强语义子域名，接入前必须先做跨项目占用核查。
2. 模板镜像的初始化路径，必须在变更前验证，不允许先改后试。
3. 一旦出现平台层异常，要优先产出“冻结 + 重规划”方案，而不是持续原地修补。
4. `mindhikers.com` 体系必须先做“首页 / CMS / 产品”子域边界设计，再做任何单项目域名接入。
5. 当跨项目异常出现时，必须同时输出“本项目结论”和“可同步给隔壁项目的排查口径”，避免双方各自猜测。
