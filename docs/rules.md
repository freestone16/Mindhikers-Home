# MindHikers Rules

更新时间：2026-03-30

## 平台与域名规则

1. 在 Railway 上操作任何自定义域名前，必须先确认当前链接项目名称、服务列表、Workspace 名称，并记录在执行上下文里。
2. 涉及共享根域名（例如 `mindhikers.com`）的子域名新增、切换、退役时，必须先做“跨项目域名占用排查”，确认不会与其他活跃产品或环境混淆。
3. 未经跨项目核对，不得直接把语义强、容易复用的子域名接入单一项目，例如 `manage.*`、`admin.*`、`api.*`；Homepage CMS 这类场景优先使用显式命名，如 `homepage-manage.*`。
4. 当一个子域名可能被多个项目历史使用过时，优先采用更显式的命名，例如 `mindhikers-manage.*`、`gc-admin.*`，直到全局归属被统一治理。
5. `mindhikers.com` 作为主根域后，其下首页与各产品必须使用彼此独立的子域名，不允许把首页入口、CMS 后台入口、产品入口混在同一子域语义下。
6. 首页域名、后台域名、产品域名需要分别建账并维护映射表，至少在施工文档中写明“哪个子域名属于哪个项目”。

## Railway 施工规则

1. 对官方模板服务做结构性调整前，必须先确认镜像初始化机制，尤其是 volume 挂载路径是否会影响首启拷贝逻辑。
2. 对运行中服务的 volume、域名、数据库关系做变更时，必须先定义回滚点，再执行变更。
3. 如果平台真实状态与方案假设不一致，必须立即暂停施工，先汇报，再重规划，不允许边猜边修。
4. 当服务命名含糊时，不允许仅凭服务名判断职责，必须结合项目名、镜像源、域名绑定和变量一起确认。

## 跨项目协作规则

1. 一个项目中的域名动作，如果可能影响同根域下的其他产品，必须明确标注“潜在跨项目影响”。
2. 如果发现问题可能外溢到隔壁项目，应先冻结当前变更，再输出可同步给隔壁项目的结论与修复建议。
3. 所有重大平台偏差都要写入 `docs/lessons.md`，避免重复踩坑。
4. 新产品接入 `mindhikers.com` 体系时，先定子域名边界，再做服务部署，不能反过来先上线服务再抢语义域名。

## Headless 缓存与回刷规则

1. WordPress webhook 发出的 revalidate tag 必须与 Next.js 实际 `fetch(..., { next: { tags } })` 使用的 tag **逐字一致**；命名约定必须集中在同一处维护，禁止 WP 与 Next.js 各自发明一套。
2. `POST /api/revalidate` 只允许做 **POST-only** 回刷，不允许暴露 GET 形式的副作用接口。
3. revalidate 接口必须校验 tag 白名单或明确前缀，禁止接受任意 tag 以免错误或恶意清空 ISR 缓存。
4. WordPress 侧 webhook 即使保持 non-blocking，也必须对“配置缺失、派发开始、请求初始化失败”至少三类事件留下可检索日志，不能静默失败。

## Headless 图片放行规则

1. 任何前台组件改为消费 WordPress 媒体库图片（如封面图、二维码、社交图）时，必须同步核对 `next.config.mjs` 中的 `images.remotePatterns` 与 CSP `img-src`，避免页面功能改完但生产环境被策略拦截。
2. 允许图片来源时优先写“明确域名/域名模式”，不要为了省事把 CSP 放宽到全量第三方来源。
