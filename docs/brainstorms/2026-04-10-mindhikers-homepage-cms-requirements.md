# Mindhikers 主页 CMS 化与信息架构重定义 — 需求文档

## 0. 文档目的

这份文档是 Mindhikers 主页从"前端硬编码 + Elementor 模板拼接"转向"基于 WordPress 的可持续 CMS"的产品需求文档（lightweight PRD）。

它要回答的问题是：

1. Mindhikers 是一个什么样的品牌、为谁服务、提供什么价值
2. 主页应该有哪些模块、每个模块承载什么内容类型
3. 这些内容如何在 WordPress 后台被结构化地管理
4. 双语（中文 / 英文）策略如何在数据层和工作流上落地
5. 本次工作的"完成"边界在哪里（拆 M1 / M2）

它不回答的问题（这些留给 `ce-plan`）：

- 具体用哪些插件、CPT slug 怎么定、字段名怎么写、API 怎么设计
- Elementor 模板如何重构、CSS 如何组织
- 部署、域名切换、数据迁移的具体技术步骤

### 2.2 核心问题

> **2026-04-12 更新**：问题 3 和 5 已通过 Unit 1–7 缓解（Hero 文案已对齐品牌定位、PRD 本身已提供锚点），但问题 1/2/4 仍然是 M1 的核心待解。

2. **后台不可用**：老卢无法通过 WordPress 后台直接管理 About / Product / Contact 等模块的内容，必须依赖 Elementor 编辑器或代码改动

4. **双语数据层缺失**：当前 `/en` 是过渡方案（独立新建页 + CSS/JS 隐藏中文主题壳），不存在真正的 ZH↔EN 翻译关系。视觉层面双语可访问，但数据层无翻译关系
5. **推进缺锚点**：~~之前的工作偏 hotfix 式 — 缺少一个稳定的 PRD 作为后续 ce-plan 和 ce-work 的依据~~ → 本 PRD 已建立，锚点问题已解决

### 2.3 这次为什么是重要的

1. 主页是 Mindhikers 这个频道的门面 — 没有清晰的信息架构，频道运营无法正常启动
2. 单人运营前提下，"老卢能在 5 分钟内自己改完一个区块的内容"是频道能持续运转的最低门槛
3. 当前的 Elementor 数据层方案如果继续往下拼，每次内容更新都会变成开发任务，无法可持续

## 3. 信息架构（Information Architecture）

### 3.1 主页五大区块（最终架构）

```
首页 (/)
├── Hero            — 频道当期主题展示位（1-2 月更新一次）
├── About           — 品牌介绍 + 主理人视角
├── Product         — 数字工具集展示位（多产品结构，CPT 列表 + Featured）
├── Blog            — 知识成长频道的内容主载体（双层分类）
└── Contact         — 静态信息 + 联系表单 + 订阅入口 + 社交矩阵
```

英文首页 (/en) 与中文首页五大区块**严格对等**，每个区块都必须有对应的英文版本。

### 3.1.1 数据源映射（信息架构 → WordPress 数据源）

```
┌────────────────────────────────────────────────────────────────────────┐
│  主页 (/  和  /en)  —  Astra 主题模板渲染                              │
└────────────────────────────────────────────────────────────────────────┘
        │
        ├──> [Hero]          ← 单个 "Hero 主题" 单例（WP Option 或单例页）
        │                       双语由 Polylang 维护
        │
        ├──> [About]         ← 单个 "About" 静态页（Gutenberg 编辑）
        │                       双语由 Polylang 维护
        │
        ├──> [Product]       ← CPT: product 列表
        │    │                   一个 Featured + 其他
        │    │                   双语由 Polylang 维护
        │    └──> 点击进入   ← /product/<slug>  或  /en/product/<slug>
        │                        （CPT 的详情页路由）
        │
        ├──> [Blog]          ← WP 原生 Posts
        │    │                   双层分类：
        │    │                     主分类 = AI / 碳硅共生 / 脑神经科学
        │    │                     次级分类 = 深度 / 速记 / 视频 / 工具
        │    │                   双语由 Polylang 维护（但 Blog 可选翻译）
        │    └──> 完整列表页  ← /blog  和  /en/blog
        │
        └──> [Contact]       ← 静态字段（WP Option 或单例页）+
                                 CPT: 社交平台条目（每条一个 row） +
                                 表单（SureForms/WPForms Lite） +
                                 订阅（SureForms/WPForms Lite）
                                 双语由 Polylang 维护（字段级）

┌────────────────────────────────────────────────────────────────────────┐
│  WordPress 后台 (wp-admin)                                             │
├────────────────────────────────────────────────────────────────────────┤
│  侧边栏入口：                                                          │
│    ▸ 仪表盘                                                            │
│    ▸ 文章（Blog）                                                      │
│    ▸ 产品（CPT: product）     ← 新建                                   │
│    ▸ 媒体库                                                            │
│    ▸ 页面（About / Hero / Contact 单例页 或者 Option 入口）            │
│    ▸ 评论（默认）                                                      │
│    ▸ 外观、插件、用户、设置、工具                                      │
│    ▸ Polylang 多语言                                                   │
│    ▸ SureForms / WPForms（表单管理 + 提交记录）                        │
└────────────────────────────────────────────────────────────────────────┘
```

### 3.2 各模块语义角色（一句话定义）

| 模块 | 语义角色 | 不是什么 |
|---|---|---|
| Hero | 频道当期主打主题 | 不是"装好就忘的设计装饰" |
| About | "Mindhikers 是什么/做什么/服务谁" | 不是"页面设计说明"或"老卢简历" |
| Product | "频道为内容创作者提供哪些数字工具" | 不是"网站特性介绍" |
| Blog | "知识成长频道的主要内容载体" | 不是"随便写点什么的私人日志" |
| Contact | "频道的对外接触面（订阅 / 合作 / 社交）" | 不是"页面底部的表单义务" |

## 4. 各模块需求详述

### 4.1 Hero 模块

**R1. Hero 的语义是"频道当期主打主题"**
- 不是普通的网页头部装饰
- 每次更新对应一次频道的运营节奏（节奏：1-2 个月一次）
- 必须支持快速更新 — 老卢自己 5 分钟内能完成一次替换

**R2. Hero 必须包含的字段**
- 当期主题名（短句，作为最大字号的标题）
- 副标题或一段不超过 80 字的描述
- 主 CTA 按钮（按钮文字 + 链接，链接通常指向某个产品页或博客文章）
- 次 CTA 按钮（可选，结构同主 CTA）
- 配图（一张图片，从 WordPress 媒体库选择）

**R3. Hero 在后台的管理方式**
- 必须能从 WordPress 后台直接修改以上字段，不依赖 Elementor 编辑器
- 字段集中在一个表单视图里，避免散落在多个地方
- 配图必须支持媒体库一键替换，不要求每次都重新上传到自定义路径

**R4. Hero 的双语要求**
- 必须支持 ZH 和 EN 两个版本一一对应
- 切换语言后，Hero 的所有字段都对应到该语言版本

### 4.2 About 模块

**R5. About 模块的语义是"品牌介绍 + 主理人视角"**
- 主体是 Mindhikers，不是老卢个人简历
- 但允许含有主理人视角的表达
- 必须呈现品牌定位的核心信念、三大支点、目标用户

**R6. About 模块的内容来源**
- 第一版直接使用品牌定位章节的"完整品牌叙述（原文）"作为底稿
- 后续可以由老卢自己迭代，但每次迭代都不能偏离品牌定位章节的关键定位词

**R7. About 模块在后台的管理方式**
- 必须能从 WordPress 后台直接编辑 About 区文案（建议用 Gutenberg 或 ACF）
- 不依赖 Elementor 编辑器

**R8. About 模块的双语要求**
- 必须 ZH+EN 严格对等
- 英文版的内容是中文版的精确翻译，不允许"英文版只放占位文字"

### 4.3 Product 模块

**R9. Product 模块的语义是"频道为内容创作者提供的数字工具集"**
- 形态明确：数字工具 / Web 产品（不是内容产品、不是方法论、不是服务）
- 当前唯一产品："黄金坩埚（Golden Crucible）"
- 长期会有多个产品 — 产品模型必须支持多产品扩展

**R10. Product 必须以"自定义文章类型（Custom Post Type）"实现**
- 每个产品是一个独立的 WordPress 后台条目
- 不允许把多个产品挤在一个 Elementor 页面或 Gutenberg 页面里
- 这条是硬约束 — 否则后续扩展会复现"内容模型混乱"问题

**R11. 每个产品必须包含的字段**
- 产品名（短）
- 副标题 / 一句话定位
- 简介（不超过 200 字，用于产品列表卡片和首页 Featured 展示）
- 完整描述（不限长度，用于产品详情页）
- 产品状态（可选枚举：构思中 / 开发中 / 公测 / 正式发布 / 已下线）
- Logo 或主视觉图（从媒体库选择）
- 产品入口链接（外链或内部页）
- 是否 Featured（布尔值；同一时刻只能有一个 Featured）

**R12. 黄金坩埚作为第一个产品的粗略描述（待老卢后续维护细化）**

> ⚠️ 以下内容是 PRD 阶段的"占位版"，不是最终版。M1 上线后由老卢在 WordPress 后台自行细化。ce-plan 阶段不应该把 R12 的具体内容当成最终文案来做实施决策。

- 形态：一套面向内容创作者（含老卢自己）的数字工具集
- 覆盖环节：选题 → 头脑风暴 → 讨论 → 视频制作 → 分发
- 服务对象：知识成长频道运营者、独立创作者
- 状态：**构思中**（PRD 落盘时）
- 详细字段由老卢后续在后台自行补充

**R12.1 重要澄清**：当前 `src/data/site-content.ts` 和 staging 前台显示的"黄金坩埚 Live now / Featured release"是从旧模板和旧版面沿用的**错误状态标签** — 它的真实状态是"构思中"。M1 内容迁移时必须纠正这个状态，不能按"Live now"迁移。

**R13. 首页 Product 区的展示规则**
- 显示 1 个 Featured 产品（大卡片）
- 显示其他产品（小卡片网格，按发布时间倒序）
- 具体展示数量（建议 3 个）由 ce-plan 阶段结合视觉设计决定
- 点击任何卡片进入对应的产品详情页

**R14. Product 详情页**
- 每个产品自动生成一个详情页（路由如 `/product/<slug>` 或 `/products/<slug>`）
- 详情页内容来自 R11 字段
- 详情页必须有 ZH 和 EN 两个版本

**R15. Product 模块的双语要求**
- 严格对等：每个产品必须 ZH+EN 一一对应
- 翻译关系通过多语言插件（如 Polylang）维护

### 4.4 Blog 模块

**R16. Blog 模块的语义是"知识成长频道的主要内容载体"**
- 是 Mindhikers 内容生产的主战场
- 必须围绕三大支点和频道定位组织
- 数据层使用 WordPress 原生 Posts（已完成）

**R17. Blog 必须使用双层分类**
- 主分类（Categories）：3 个固定值
  - AI 技术
  - 碳硅共生
  - 脑神经科学
- 次级分类（用 WordPress Categories 的子分类，或独立 Taxonomy）：
  - 深度
  - 速记
  - 视频
  - 工具
- 每篇文章在写作时必须同时选一个主分类 + 一个次级分类

**R18. Blog 不强制使用标签**
- 短期不引入额外的 tag 体系
- 如果未来内容多到双层分类不够用，再考虑引入标签

**R19. Blog 文章必须包含的字段**
- 标题
- 主分类 + 次级分类
- 摘要 / 引言（用于列表卡片和 SEO description）
- 正文（Gutenberg 编辑器）
- 封面图
- 发布时间
- 作者（短期固定为老卢，但保留 WordPress author 字段）

**R20. Blog 的双语要求（与其他模块不同 — 这是双语策略的例外）**
- 默认每篇文章只有中文版
- 老卢可以"精选"某些文章翻译成英文版
- 数据层支持 ZH↔EN 翻译关系（用 Polylang），但工作流不强制配对
- 英文 Blog 列表只显示已翻译的文章

**R21. 首页 Blog 区的展示规则**
- 显示最近 N 篇文章（N 由 ce-plan 阶段决定，建议 3-6 篇）
- 列表按发布时间倒序
- 每条显示：标题、摘要、主分类、次级分类、发布时间、封面图
- "查看全部文章"链接进入完整 Blog 列表页

### 4.5 Contact 模块

**R22. Contact 模块包含三类元素**

| 元素 | 必有 |
|---|---|
| 静态联系信息 | ✓ |
| 联系表单 | ✓ |
| 订阅入口 | ✓ |
| 社交矩阵 | ✓ |

**R23. 静态联系信息字段**
- 主邮箱：`ops@mindhikers.com`（替换当前的 `contactmindhiker@gmail.com`）
- 坐标：Shanghai / Remote
- 这两个字段必须可后台修改

**R24. 联系表单**
- 使用 staging 已装的 SureForms 或 WPForms Lite（具体选哪个由 ce-plan 决定）
- 字段：姓名 + 邮箱 + 主题 + 内容
- 提交后存到 WordPress 后台，可以从后台查看
- 必须有"提交成功"提示
- 必须有防垃圾机制（reCAPTCHA 或 honeypot，由 ce-plan 决定）
- 表单提交不强制要求邮件转发（短期内老卢从后台查看即可）

**R25. 订阅入口**
- 字段：仅邮箱（短期最简实现）
- 提交后存到 WordPress 后台
- 不强制接邮件服务（如 Buttondown / Mailchimp / ConvertKit）— 这是 M3 之后的事
- 必须有"订阅成功"提示
- 订阅成功后短期不需要双重确认（double opt-in），M3 之后再考虑

**R26. 社交矩阵**
- 必须展示的平台（短期）：
  - Twitter / X
  - Bilibili
  - 微信公众号
- 微信公众号的展示形态特殊 — 通常用二维码图片，不是 URL
- 后台必须支持上传二维码图片（一张图片字段）
- 其他平台用 URL + 平台 logo 展示
- 后台必须支持新增 / 删除 / 排序社交平台条目

**R27. Contact 模块的双语要求**
- 静态信息（邮箱、坐标）双语共享
- 表单字段标签和提示文字必须双语
- 社交矩阵的平台名称双语（中文版显示"微信公众号"，英文版显示"WeChat Official Account"）

### 4.6 顶部导航 + Footer

**R28. 顶部导航**
- 中文：关于 / 产品 / 博客 / 联系
- 英文：About / Product / Blog / Contact
- 必须有清晰的语言切换入口（ZH ↔ EN）
- 当前 staging 已完成，但要确认进入新架构后导航仍指向正确位置

**R29. Footer**

- 主邮箱（与 Contact 区同源）
- 社交矩阵（与 Contact 区同源 — 同一份数据，两处展示）
- 版权行（已完成中文化）
- 必须有清晰的语言切换入口

## 5. 双语策略（Bilingual Strategy）

### 5.1 总原则

| 内容类型 | 双语要求 | 维护成本 |
|---|---|---|
| Hero | 严格对等 | 低（1-2 月更新一次） |
| About | 严格对等 | 低（不常改） |
| Product 列表（首页区块） | 严格对等 | 低（短期产品少） |
| Product 详情页 | 严格对等 | 中（每个产品都要英文版） |
| Contact | 严格对等 | 低 |
| Footer / 导航 | 严格对等 | 低 |
| Blog 文章 | **精选翻译**（不强制） | 由老卢决定 |

### 5.2 关键决策

- **使用多语言插件维护翻译关系** — 推荐 Polylang（免费版即可）
  - 理由：Polylang 支持 CPT 翻译、可选翻译（不强制）、与 Astra/Elementor 兼容好
  - WPML 也是选项，但收费且更重 — 留给 ce-plan 评估
- **所有 CPT 都启用 Polylang** — 包括将来引入的产品 CPT
- **Blog 的"精选翻译"在数据层和工作流的体现**：
  - 数据层：每篇文章都可以有 ZH/EN 翻译关系
  - 工作流：写作时不强制创建英文版
  - 前台：英文 Blog 列表只显示有英文版本的文章

### 5.3 不允许的状态

- ❌ 英文版只放占位文字（比如 "Coming soon"）
- ❌ 英文版用机器翻译（粗糙的机翻不符合品牌门面定位）
- ❌ 中英混排但没有明确语言意图（比如英文页面里突然出现一段中文）
- ✅ 允许保留：品牌名、产品名（如 黄金坩埚 / Golden Crucible 同时使用）、邮箱、URL、技术名词

## 6. 后台管理体验要求（CMS UX Requirements）

### 6.1 老卢能力假设

- 老卢是技术背景的主理人，能用 WordPress 后台、能上传图片、能编辑 Gutenberg 块
- 但老卢的时间宝贵 — 后台 UX 必须围绕"5 分钟完成一次内容更新"设计
- 老卢不愿意进 Elementor 编辑器改文字（这是当前痛点）

### 6.2 关键 UX 要求

**R30. 后台编辑入口必须清晰**
- 在 WP Admin 侧边栏能直接看到 "Hero 主题位 / 产品 / 博客 / Contact 信息" 等入口
- 不需要"先打开页面 → 再打开 Elementor → 再找到某个 widget → 再点击某个文字"这种 4 步流程
- 理想状态：进入后台 → 1 次点击进入对应模块的编辑界面

**R31. 字段必须围绕语义命名，不能用 Elementor 风格的技术字段**
- ✅ "当期主题名"、"产品状态"、"主分类"
- ❌ "Section_3_Title"、"Widget_block_2_text"

**R32. 媒体管理统一走 WordPress 媒体库**
- 不允许任何字段使用外链 URL 引用图片（除非是社交平台 logo 这种通用资源）
- 上传图片应该在媒体库里有清晰的命名和分类

**R33. 双语切换必须一目了然**
- 后台编辑某个内容时，必须能清楚看到当前是 ZH 还是 EN
- 切换语言版本不超过 1 次点击
- 翻译关系（ZH ↔ EN）必须可见且可手动绑定/解绑

**R34. 错误友好**
- 误删除应该有 Trash 兜底（WordPress 默认能力，确保启用）
- 重要变更应该有版本历史（WordPress 默认能力，确保启用）

## 10. 关键决策与理由（Key Decisions）

| 决策 | 理由 |
|---|---|
| Product 必须用 CPT 实现 | 解决当前"产品 = 网站特性堆"的根本混乱；为长期多产品扩展铺路；强制把"产品"和"网站特性"在数据层分清 |
| About 内容用品牌定位原文打底 | 老卢已经写出了清晰的品牌叙述，没必要再造一份；把它直接落进 About 是最快也最忠于品牌定位的方式 |
| Blog 用双层分类（支点 × 形式） | 三大支点呼应品牌定位，4 种形式呼应内容生产实际；每篇文章必须同时归位，杜绝"无分类乱放" |
| Hero 升级为"当期主题展示位" | 将 Hero 从"装好就忘的设计装饰"升级为有运营语义的内容位，呼应频道运营节奏 |
| 双语策略：首页五区块严格对等 + Blog 精选翻译 | 品牌门面必须完整双语；写作压力大的 Blog 不被对等翻译拖住节奏；给老卢留出选择权 |
| 装 Polylang 维护翻译关系 | Polylang Free 版能力足够；与 Astra/Elementor 兼容好；比 WPML 轻 |
| 本次只做单人权限 | 老卢现在是单兵模式；过早设计协作流是常见浪费；WordPress 默认能力支持未来升级 |
| 拆 M1 + M2 + 分两次 ce-plan | M1 后会有新认知，M2 的实施方案应该在 M1 完成后再写，避免一次性吞下太大方案被锁死 |
| Hero 配图走媒体库 | 避免外链 URL 引用，统一媒体管理 |
| 微信公众号用图片字段（二维码） | 微信公众号没有公开 URL，必须用二维码图片 |
